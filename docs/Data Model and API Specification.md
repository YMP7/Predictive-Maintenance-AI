# Data Model and API Specification

> [!IMPORTANT]
> **Implementation Status (June 21, 2026):**
> - **Current Implemented Specification**: Runs locally using **SQLite** for database storage and a set of synchronous **REST API endpoints** served by FastAPI on port 8000.
> - **Target Specification (Roadmap)**: The NoSQL (Firebase) schema, batch telemetry ingestion, MQTT/WebSockets communication, and Bhashini translation API are design specifications for the future production system.

---

### 1. Current Implemented Specification (SQLite & FastAPI)

The prototype uses a local relational SQLite database (`data/digital_twin.db`) and a pull-based REST API for dashboard queries.

#### 1.1 SQLite Schema

##### Table: `telemetry`
Stores historical sensor readings and classified status.
```sql
CREATE TABLE IF NOT EXISTS telemetry (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    machine_id TEXT,
    timestamp TEXT, -- ISO-8601 UTC format (Z)
    vibration_x REAL,
    vibration_y REAL,
    vibration_z REAL,
    vibration_rms REAL,
    temperature REAL,
    current REAL,
    status TEXT -- 'Normal', 'Warning', 'Critical'
);
```

##### Table: `alerts`
Stores logged alerts generated by the AI agent.
```sql
CREATE TABLE IF NOT EXISTS alerts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT,
    machine_id TEXT,
    type TEXT, -- e.g. 'Fault Detection', 'RUL Warning'
    severity TEXT, -- 'Low', 'Medium', 'High', 'Critical'
    message TEXT,
    fault_type TEXT -- 'Normal', 'Bearing Wear', 'Misalignment', 'Overheating', 'Electrical Fault'
);
```

#### 1.2 Implemented REST API Endpoints

All endpoints serve on `http://localhost:8000` (or `API_PORT`).

| Endpoint | Method | Description | Sample Response |
| :--- | :--- | :--- | :--- |
| `/health` | `GET` | System health check. | `{"status":"healthy","timestamp":"2026-06-21T08:00:00Z"}` |
| `/api/summary` | `GET` | Dashboard overall summary. | `{"total_machines":4,"total_alerts":12,"average_rul_days":24.5,...}` |
| `/api/machines` | `GET` | Retrieves metadata of all machines. | `[{"machine_id":"M001","name":"Machine M001","type":"Lathe",...}]` |
| `/api/machines/{machine_id}/status` | `GET` | Retrieves the latest status and RUL. | `{"machine_id":"M001","status":"Normal","rul_days":30,...}` |
| `/api/machines/{machine_id}/telemetry` | `GET` | Retrieves telemetry history (limit query). | `[{"machine_id":"M001","timestamp":"...","temperature":48.2,...}]` |
| `/api/machines/{machine_id}/trends` | `GET` | Returns mean, min, max metrics. | `{"vibration":{"mean":0.8,"max":1.4},"temperature":{...}}` |
| `/api/machines/{machine_id}/alerts` | `GET` | Returns historical alerts for one machine. | `[{"timestamp":"...","type":"Fault Detection","severity":"High"}]` |
| `/api/alerts` | `GET` | Returns recent alerts across all machines. | `[{"machine_id":"M001","type":"Fault Detection",...}]` |
| `/api/simulation/start` | `POST` | Starts real-time simulation thread. | `{"status":"simulation started","interval":1.0}` |
| `/api/simulation/stop` | `POST` | Stops real-time simulation thread. | `{"status":"simulation stopped"}` |
| `/api/machines/{machine_id}/fault` | `POST` | Injects a specific fault pattern. | `{"status":"fault injected","fault_mode":"bearing_wear"}` |

### 2. Target Specification (Roadmap NoSQL Schema)

The enterprise cloud-hosted version is designed to utilize a document-oriented NoSQL database (e.g. Firebase) for cross-factory scalability.

#### 2.1 `machines` Collection
| Field | Type | Description |
| :--- | :--- | :--- |
| `machine_id` | String | Unique Identifier (e.g., node-001-lathe). |
| `name` | String | User-friendly name. |
| `type` | String | Category (Motor, Lathe, Pump). |
| `thresholds` | Object | Custom sensor limits for alerts. |
| `location` | String | Factory section or GPS coordinates. |

#### 2.2 `telemetry` Collection (Time-Series)
| Field | Type | Description |
| :--- | :--- | :--- |
| `machine_id` | String | Reference to the machine. |
| `timestamp` | Number | Unix timestamp of reading. |
| `vibration` | Array | [x, y, z] values in Gs. |
| `temperature` | Number | Surface temperature in Celsius. |
| `current` | Number | Amperage draw. |
| `status` | String | Predicted state (Normal/Warning/Error). |

### 3. Target API Endpoints (Roadmap)

#### 3.1 Telemetry Ingestion (`POST /api/v1/telemetry`)
Designed for edge devices to push batches of sensor data.
```json
{
  "machine_id": "M001",
  "data": [
    {"ts": 1717490000, "v": [0.1, 0.2, 1.0], "t": 45.5, "c": 2.1}
  ]
}
```

#### 3.2 Prediction Retrieval (`GET /api/v1/prediction/{machine_id}`)
Retrieves ML-model predictions from cloud services.
```json
{
  "machine_id": "M001",
  "rul_days": 14,
  "confidence": 0.89,
  "fault_type": "Bearing Wear",
  "recommended_action": "Lubricate bearings within 48 hours"
}
```

#### 3.3 Localization Service (`POST /api/v1/alerts/translate`)
Internal translation service using Bhashini API.
- **Payload**: `{"text": "Check motor temperature", "target_lang": "hi"}`

---

### 4. Target Communication Protocols (Roadmap)

- **MQTT**: Lightweight publish-subscribe messaging for high-frequency telemetry upload over topics like `factory/site_id/machine_id/telemetry`.
- **WebSockets**: Persistent full-duplex connection for instantaneous Digital Twin updates on the React dashboard.
- **HTTPS**: Secure REST queries for configurations and admin controls.

---

### 5. Target Data Retention Policy (Roadmap)

- **Real-time Data**: Granular high-frequency raw sensor waveforms kept for 7 days.
- **Aggregated Data**: Hourly statistics and trends retained for 1 year.
- **Alert Logs**: Alert records preserved indefinitely for audit trails and maintenance histories.
